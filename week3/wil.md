### 3. 정리

**1) 더 편하게, 더 간결하게 하고 싶어**

- 기존 HTML + JavaScript 방식은 규모가 커질수록 협업과 유지보수가 어려워진다.  
  - 예를 들어, 여러 명이 하나의 HTML 파일(수백~수천 줄)을 수정하면 코드 충돌이 발생하기 쉽다.  
  - 기능을 추가하거나 수정할 때, 코드의 다른 부분까지 모두 파악해야 한다.
- 이를 해결하기 위해 코드 구조를 나누고 재사용성을 높이는 방법이 필요했다.

---

**2) 해결사 등장! (두둥탁)**

- React가 이러한 문제를 해결한다.  
  → 화면을 컴포넌트 단위로 나누어 관리하는 개념 도입
- `npx create-react-app (폴더명)` 명령어를 통해 기본 React 프로젝트를 생성할 수 있다.
  - `npm start` 로 실행 시, React로 구성된 페이지가 브라우저에서 동작함.
- React는 HTML을 직접 다루지 않고 JS로 모든 화면을 렌더링한다.  
  - 각 기능(화면 단위)을 컴포넌트로 나누어 개발한다.
  - 예: `Me.js`, `Father.js`, `Dog.js` 같은 독립적인 컴포넌트를 만들어 `App.js`에 조합.
- 이렇게 하면 여러 명이 동시에 작업하더라도 충돌이 줄고 코드 재사용성과 생산성이 향상된다.  

- React의 핵심 개념:
  - **JSX(JavaScript XML)** : 자바스크립트 코드 안에서 HTML 문법을 그대로 쓸 수 있는 확장 문법.  
    → JS와 HTML을 결합하여 더 간결한 UI 코드 작성 가능.
  - **Component** : 재사용 가능한 UI 단위.  
    → 화면을 여러 개의 독립적인 컴포넌트로 분리해 관리.
  - **SPA (Single Page Application)** :  
    → 한 번 로딩된 페이지 안에서 필요한 부분만 갱신하는 방식.  
      새로고침 없이도 빠른 전환이 가능해 앱처럼 작동한다.

---

**3) The REACT SAGA**

- React는 UI를 효율적으로 구축하기 위한 자바스크립트 라이브러리다.  
  - 라이브러리는 필요한 기능을 가져다 쓰는 개념이며,  
    프레임워크는 정해진 틀과 규칙 내에서 개발하는 방식이다.  
  - 프론트엔드의 React는 라이브러리, 백엔드의 Spring/Django는 프레임워크의 대표적인 예다.

### 4. 느낀점
- 짱구 페이지로 왜 React를 사용해야 하는지 바로 알았다.  
- 컴포넌트 단위로 화면을 구성하는 것이 훨씬 효율적인 것이 바로 나타났다.
- JSX 문법이 지금 간단하게 해서 괜찮아 보이지만 조금 더 들어가면 어려울 것으로 예상된다.
